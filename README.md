# SENSE-LINK · Edge-AI Communications Network

> **iQOO Hackathon 2026 — "Red Light, Green Light"** · *Project Monster Transmit*
> A 100% on-device, offline edge-AI app that removes communication barriers between
> **Deaf / mute** and **hearing** people. Built to run natively on the iQOO phone.

---

## What it does

- **Deaf/mute → hearing:** sign over the camera → on-device hand tracking → gesture tokens →
  a context engine composes a fluent sentence → the phone **speaks it aloud**.
- **Hearing → Deaf/mute:** speak → offline speech-to-text → text on screen + **haptic buzz**
  (relayed to a smartwatch in the full vision).
- **Shake to clear.** Everything runs locally — works in **airplane mode**.

It fixes the three failures of existing tools: broken literal "Tarzan text", cloud dependency
(dead in metros/elevators/crowds), and phone overheating from continuous CV + cellular radio.

## On-device pipeline (what's real)

- **Camera** → `getUserMedia` (rear/front).
- **Hand tracking** → MediaPipe Hand Landmarker, 21 3D landmarks, **GPU delegate**, bundled
  WASM — measured inference latency shown live.
- **Gesture classifier** → Euclidean distance + finger-extension geometry → semantic tokens.
- **Context composer** → rewrites token fragments into fluent sentences *(deterministic
  on-device stand-in for the Gemma-3 1B INT4 layer described in [docs/PROJECT_SPEC.md](docs/PROJECT_SPEC.md))*.
- **Text-to-Speech / Speech-to-Text** → Web Speech API (locale, voice profile, rate wired to config).
- **Shake-to-clear** → DeviceMotion accelerometer · **Haptics** → `navigator.vibrate`
  *(phone stands in for the smartwatch BLE GATT link)*.

**Demo-safety:** every gesture is also a tappable chip, so the operator can drive the exact
same pipeline by tapping if lighting/camera is poor on stage. The app never hard-fails.

## How the offline WebView works

The web app and all of its assets are bundled under
`android/app/src/main/assets/www/` and served over a **secure virtual https origin**
(`https://appassets.androidplatform.net`) via `WebViewAssetLoader` — not `file://`.
This is what makes ES-module imports, **WebAssembly streaming**, and `getUserMedia`
(secure-context) all work from local files. A custom path handler fixes the MIME types
(`.mjs`/`.js` → `text/javascript`, `.wasm` → `application/wasm`) that the default asset
handler gets wrong, which would otherwise silently break MediaPipe.

The camera works inside the WebView via `WebChromeClient.onPermissionRequest { grant(...) }`
plus a runtime `CAMERA` + `RECORD_AUDIO` request — see
[`MainActivity.kt`](android/app/src/main/java/com/kraftshala/senselink/MainActivity.kt).

## On-device testing & fixes

Tested by sideloading the APK on **3 physical iQOO phones**.

- **Fixed — "Microphone permission denied" on the onboarding mic toggle.** Root cause: the
  WebView's `onPermissionRequest` was granting the web-layer media request *blindly*, without
  ensuring the **OS-level** `RECORD_AUDIO` permission was actually held — so `getUserMedia({audio:true})`
  was denied downstream on every device. Fix: `onPermissionRequest` now grants only when the
  matching OS permission is present, and otherwise **requests it on demand** and grants once the
  user approves (see `handleWebPermissionRequest` in
  [`MainActivity.kt`](android/app/src/main/java/com/kraftshala/senselink/MainActivity.kt)).
  The same path makes the camera permission robust.
- Added `FLAG_KEEP_SCREEN_ON` so the screen never sleeps mid-demo.
- Made the build portable (removed the machine-specific hardcoded JDK path from `gradle.properties`).

> **Known WebView limitation (roadmap item):** the *reverse* speech-to-text loop uses the Web
> Speech API (`webkitSpeechRecognition`), which Android System WebView does not implement (it only
> exists in Chrome-the-browser). It degrades gracefully today (a toast, no crash). The fix is the
> native `SpeechRecognizer` bridge (planned for the next iteration). Text-to-Speech (the headline "speaks aloud")
> uses `speechSynthesis`, which **is** available in WebView.

## On-device LLM — Gemma-2B (4-bit) via LiteRT &nbsp;✅ implemented

The composer is now backed by a **real on-device LLM**.
[`LlmBridge.kt`](android/app/src/main/java/com/kraftshala/senselink/LlmBridge.kt) runs Google
**Gemma-2B (int4)** through the **LiteRT / MediaPipe LLM Inference** runtime and is exposed to the
web app as `window.AndroidLLM`. `composeAndSpeak()` calls it with a low-temperature prompt that
rewrites the gesture tokens into one fluent sentence; the source tag shows **GEMMA·2B** when the
model is live. It runs fully locally — airplane-mode safe.

**The model is not shipped in the APK** (it's ~1.3 GB). Provision it once, either way:

```bash
# A) Side-load for the demo (most reliable):
adb push gemma.task /sdcard/Android/data/com.kraftshala.senselink/files/gemma.task
#    then relaunch, or System Config -> MODEL HOT-RELOAD.

# B) First-run download: set window.LLM_MODEL_URL (a direct .task URL) in index.html,
#    then MODEL HOT-RELOAD streams it into filesDir/llm/gemma.task and auto-initialises.
```

Get a compatible `.task` from Google AI Edge / Kaggle / the `litert-community` HuggingFace org
(accept the Gemma license). *Format note:* GGUF is llama.cpp's format; the LiteRT/MediaPipe path
uses a converted int4 `.task` bundle — a llama.cpp + GGUF (JNI) engine is a drop-in alternative.

**Demo-safety:** if no model is present (or it loads slowly), the app transparently uses the
deterministic composer, and a 6 s watchdog guarantees the demo never hangs on the model — the
source tag then reads **GEMMA·INT4**.

## Roadmap — next feature (idea + solution)

### Universal smartwatch bridge (BLE + LLM-generated protocol)

**Idea.** When the app pairs with **any** smartwatch over Bluetooth, identify the watch and have the
on-device LLM generate the right communication payload so SENSE-LINK can relay the finalized
sentence + haptic pulses (double = sentence start, long = complete) to that specific watch — without
hand-coding a driver per brand.

**Solution.**
1. **Discover + identify.** BLE scan/connect (`BluetoothLeScanner` → `BluetoothGatt`), then read the
   standard **Device Information Service** (`0x180A`): **Manufacturer Name** (`0x2A29`) and
   **Model Number** (`0x2A24`) → brand + model string.
2. **LLM-generated adapter.** Feed `"<brand> <model>"` to the on-device Gemma model with a prompt
   that returns a structured descriptor: target **service/characteristic UUIDs**, the text-notify
   byte payload format, and the haptic-pattern mapping. (Standards-first: target the **Alert
   Notification Service** `0x1811` / Nordic UART where exposed; proprietary/encrypted watches still
   need vendor SDKs — documented honestly.)
3. **Relay.** Write the sentence + haptic metadata to the resolved characteristic. The phone's
   existing `navigator.vibrate` path already stands in for the wrist haptic, so the UI/UX is unchanged
   — the BLE link simply becomes real.

These two features are scoped for the next iteration; the current deliverable is the working,
mic-fixed, fully-offline WebView APK in this repo.
